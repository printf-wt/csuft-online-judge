## 二、一层数据流图（Level-1 DFD）

### 处理过程说明

| 编号 | 处理过程名称 | 输入数据流 | 输出数据流 | 功能描述 |
|------|------------|-----------|-----------|---------|
| 1.0 | 用户认证与管理 | 注册/登录信息 | JWT/凭证 | 用户注册、登录、JWT认证、个人资料管理、刷题统计 |
| 2.0 | 题目检索与管理 | 浏览/搜索条件、题目管理指令 | 题目列表、题面数据 | 题库CRUD、按难度/标签分类检索、极速搜索 |
| 3.0 | 代码在线编辑与提交 | 提交代码及输入 | 提交记录 | Monaco Editor代码编辑、多语言切换、一键提交 |
| 4.0 | 异步代码评测 | 提交记录、测试用例 | 评测状态、AC统计 | 异步编译运行、测试用例比对、超时拦截 |
| 5.0 | 成绩统计与排行 | 用户刷题统计 | 排行列表 | AC题数统计、通过率计算、动态排序分页 |

### 数据存储说明

| 编号 | 数据存储名称 | 说明 | 关联的处理过程 |
|------|------------|------|--------------|
| D1 | User 表 | 存储用户基本信息、刷题统计 | 1.0(读写)、4.0(更新AC数)、5.0(读取统计) |
| D2 | Problem 表 | 存储题目信息、题面描述 | 2.0(读写) |
| D3 | Submission 表 | 存储提交记录、评测结果 | 3.0(写入)、4.0(读写) |
| D4 | TestCase 表 | 存储测试用例输入输出 | 4.0(读取) |

                           │          │

         ┌──────────────┐                ┌──────────────┐
         │ (TestCase表) │                │   (User表)   │

### 子处理过程说明

| 编号 | 处理过程名称 | 输入 | 输出 | 功能描述 |
|------|------------|------|------|---------|
| 4.1 | 提交接收与入队 | 提交记录 | 评测任务 | 将提交推入线程池阻塞队列，状态设为PENDING |
| 4.2 | 代码编译处理 | 源代码+语言 | 编译结果 | 在安全目录下编译代码，捕获编译错误 |
| 4.3 | 代码运行执行 | 编译后程序+测试输入 | 运行输出 | 执行代码，监控时间和内存，超时强制终止 |
| 4.4 | 输出结果比对 | 实际输出+期望输出 | 评测结果 | 逐字节比对，忽略行末空格和文末换行 |

## 四、PowerDesigner 操作步骤

### 方法一：导入SQL生成PDM（推荐）

1. 打开 PowerDesigner 16.5
2. 菜单: `File` → `Reverse Engineer` → `Database`
3. 选择目标数据库: `MySQL 5.0` 或 `Generic ODBC`
4. 选择 `Using script file`，浏览选择 `CSUFT_OJ_PDM_DDL.sql`
5. 点击 `确定`，PowerDesigner 自动生成物理数据模型
6. 菜单: `Database` → `Generate Database` 可导出SQL

### 方法二：打开CDM概念模型

1. 打开 PowerDesigner 16.5
2. 菜单: `File` → `Open`
3. 文件类型选择 `Conceptual Data Model (*.cdm)`
4. 选择 `CSUFT_OJ_CDM.cdm` 文件打开
5. 可在图形界面中拖拽调整实体位置

### 方法三：手动创建数据流图 (BPM)

PowerDesigner 中数据流图使用 **Business Process Model (BPM)** 绘制：

1. 菜单: `File` → `New` → `Business Process Model`
2. 选择 `Data Flow Diagram` 作为 BPM 类型
3. 从左侧面板拖拽图形元素：
   - **External Entity (外部实体)**: 拖入画布，命名为"用户/学生"、"管理员"
   - **Process (处理过程)**: 拖入画布，编号命名如"1.0 用户认证与管理"
   - **Data Store (数据存储)**: 拖入画布，命名为"User表"、"Problem表"等
   - **Data Flow (数据流)**: 连接各元素，标注数据流名称
4. 按照本文档中的一层数据流图布局绘制
5. 双击元素可编辑属性和添加描述

## 五、E-R 图对照

根据需求文档中的E-R图，本系统的实体关系如下：

```
┌─────────┐         ┌─────────────┐         ┌───────────┐
│  USER   │  1   N  │  SUBMISSION │  N   1  │  PROBLEM  │
│         │────────>│             │<────────│           │
│ id (PK) │         │ id (PK)     │         │ id (PK)   │
│ username│         │ user_id(FK) │         │ title     │
│ pass_hash│        │ problem_id  │         │ difficulty│
│ email   │         │ code        │         │ time_limit│
│ role    │         │ language    │         │ mem_limit │
│ solved  │         │ status      │         │           │
│ submit  │         │ time_used   │         │           │
└─────────┘         │ memory_used │         └─────┬─────┘
                    │ created_at  │               │
                    └─────────────┘               │ 1   N
                                                  │
                                            ┌─────┴─────┐
                                            │  TESTCASE │
                                            │           │
                                            │ id (PK)   │
                                            │ problem_id│
                                            │ input_data│
                                            │ expected  │
                                            │ is_sample │
                                            └───────────┘
```

### 关系说明

| 关系 | 类型 | 说明 |
|------|------|------|
| USER → SUBMISSION | 一对多 (1:N) | 一个用户可以有多条提交记录 |
| PROBLEM → SUBMISSION | 一对多 (1:N) | 一道题目可以被多次提交 |
| PROBLEM → TESTCASE | 一对多 (1:N) | 一道题目包含多个测试用例 |
