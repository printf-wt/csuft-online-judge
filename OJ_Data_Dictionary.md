# 5. 系统数据字典 (Data Dictionary)

数据字典是对数据流图（DFD）中出现的数据流、数据存储、处理过程进行详细定义与描述的工具，是系统设计和数据库建模的重要依据。本数据字典分为 **数据流字典**、**数据存储字典** 和 **数据处理字典** 三个部分。

---

## 5.1 数据流字典 (Data Flow Dictionary)

数据流字典定义了系统内流动的各个数据结构。

| 数据流编号 | 数据流名称 | 来源 (Source) | 去向 (Destination) | 数据流组成 (Data Structure) | 简要描述 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **F01** | 注册与登录信息 | 外部实体 (E1/E2/E3) | P1.1 / P1.2 | `username` + `password` + `email` + `student_no` | 用户注册或进行身份登录时输入的初始凭证数据。 |
| **F02** | 登录凭证 JWT | P1.2 | 外部实体 (E1/E2/E3) | `token_string` (由 Header, Payload, Signature 组成，包含 `userId`, `username`, `role`, `exp` 等字段) | 系统认证成功后分发给客户端的签名会话令牌，用于后续请求 API 的身份校验。 |
| **F03** | 题目管理数据 | 外部实体 (E2/E3) | P2.3 | `title` + `description` + `input_format` + `output_format` + `difficulty` + `time_limit` + `memory_limit` | 教师或管理员在后台出题页面录入的题目元数据。 |
| **F04** | 测试用例数据 | 外部实体 (E2) | P2.4 | `List<InputFile, OutputFile>` (多组输入输出文件，压缩为 `.zip` 格式) | 教师上传的用于判定代码正确性的标准输入和期望输出数据对。 |
| **F05** | 题目列表及详情 | D2 / P2.1 / P2.2 | 外部实体 (E1/E2) | `id` + `title` + `difficulty` + `ac_rate` + `content` + `time_limit` + `memory_limit` | 展示在做题列表及做题详情页面的结构化题目描述与限制。 |
| **F06** | 代码提交数据 | 外部实体 (E1) | P3.2 | `userId` + `problemId` + `source_code` + `language` | 学生在 Monaco 编辑器编写完毕后触发提交生成的评测实体包。 |
| **F07** | 初始提交记录 | P3.2 | D3 | F06 字段 + `status: "PENDING"` + `submit_time` | 插入提交记录表中，标记当前记录等待队列消费。 |
| **F08** | 待评测记录 | D3 | P5.1 | `submissionId` + `source_code` + `language` + `problemId` | 判题调度守护线程轮询拉取到的待判题任务基本信息。 |
| **F09** | 题目时空限制 | D2 | P5.3 | `time_limit` (毫秒) + `memory_limit` (KB) | 判题引擎配置沙箱子进程的最大 CPU 时间和物理内存开销硬限制阈值。 |
| **F10** | 测试点输入数据 | P5.4 | P5.3 | 字符流 / 测试用例输入文本 (`.in`) | 重定向喂给受监控沙箱子进程的标准输入流。 |
| **F11** | 测试点期望输出 | P5.4 | P5.5 | 字符流 / 测试用例标准答案文本 (`.out`) | 正确解题运行的输出，用作判断用户程序实际输出是否正确的比对标尺。 |
| **F12** | 实际输出数据 | P5.3 | P5.5 | 用户程序控制台实际标准输出流 + `time_used` + `memory_used` | 沙箱子进程执行后捕获的控制台控制数据和其运行开销物理指标。 |
| **F13** | 评测判定结果 | P5.5 | P5.6 | `sub_status` (AC/WA/RE/TLE/MLE) + `max_time` + `max_memory` | 比对模块逐字节比对完实际输出与期望输出后得出的单点评测状态。 |
| **F14** | 评测结果与指标 | P5.6 | D3 | `status` (AC/WA/CE/TLE/MLE/RE) + `time_used` + `memory_used` + `error_log` | 将该次代码提交的汇总判题结论与详细编译运行轨迹写写回库的数据。 |
| **F15** | 统计更新数据 | P5.6 | D1 / D2 | `userId` + `problemId` + `isFirstTimeAC: boolean` | 触发用户 AC 数量加一，以及题目通过率统计重新计算的触发流。 |
| **F16** | 系统监控数据 | P6.1 | 外部实体 (E3) | `cpu_load` + `memory_load` + `thread_count` + `queue_length` | 返回给管理员的可视化系统底层物理运行健康数据。 |
| **F17** | 比赛发布与编辑 | 外部实体 (E2/E3) | P4.1 | `title` + `start_time` + `end_time` + `rules_type` + `problems_list` | 教师或管理员配置的竞赛基本元信息及题目关联绑定。 |
| **F18** | 比赛排行数据 | D5 / D3 / P4.3 | 外部实体 (E1/E2) | `List<{rank, username, solved_count, penalty_time, details}>` | 按照 ACM 罚时或 IOI 积分规则计算出的竞赛内部成绩实时排名。 |
| **F19** | 公告通知数据 | D6 / P4.4 | 外部实体 (E1/E2/E3) | `id` + `title` + `content` + `author` + `created_at` | 全站渲染或管理端编辑的公告文本内容。 |
| **F20** | 操作审计日志 | Logs / P6.2 | 外部实体 (E3) | `log_id` + `userId` + `action` + `ip` + `timestamp` | 系统关键敏感操作（如修改权限、删除题目）的历史追溯记录。 |

## 5.2 数据存储字典 (Data Store Dictionary)

数据存储字典定义了系统内物理或逻辑存储的数据文件与数据库结构。

| 存储编号 | 存储名称 | 物理媒介 | 描述 | 流入数据流 | 流出数据流 | 数据结构 / 核心字段 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **D1** | 用户库 | MySQL (User表) | 存储全站用户的账户基本信息、安全凭证与全局成绩汇总数据。 | 新增用户数据、权限修改项、累加AC与提交总数。 | 用户基本信息、登录凭证校验项、排行与统计项。 | `id` (INT, PK), `username` (VARCHAR), `password_hash` (VARCHAR), `email` (VARCHAR), `role` (VARCHAR), `solved_count` (INT), `submit_count` (INT) |
| **D2** | 题目库 | MySQL (Problem表) | 存储题库的基本属性、Markdown 描述及全局答题统计。 | 教师录入/修改的题目属性，评测更新的题目AC/提交总数。 | 题目详情与列表数据、题目时空限制值、题目统计项。 | `id` (INT, PK), `title` (VARCHAR), `description` (TEXT), `input_format` (TEXT), `output_format` (TEXT), `difficulty` (VARCHAR), `time_limit` (INT), `memory_limit` (INT), `accepted_count` (INT), `submitted_count` (INT) |
| **D3** | 提交记录库 | MySQL (Submission表) | 存储用户历次提交的代码、评测结论、时空开销及编译日志。 | 初始代码提交记录（Pending）、判题引擎最终输出的评测状态。 | 待评测记录、历史提交详情、代码统计项。 | `id` (INT, PK), `user_id` (INT, FK), `problem_id` (INT, FK), `code` (TEXT), `language` (VARCHAR), `status` (VARCHAR), `time_used` (INT), `memory_used` (INT), `error_log` (TEXT), `created_at` (DATETIME) |
| **D4** | 测试数据仓库 | 文件系统 (TestCase目录) | 存储每道题目关联的评测输入与输出文本用例文件。 | 教师上传的各测试点用例文件。 | 评测时拉取的测试点输入与标准输出期望值。 | 路径结构：`problem_id/`<br>- `<case_num>.in` (输入文本)<br>- `<case_num>.out` (期望输出文本) |
| **D5** | 比赛库 | MySQL (Contest相关表) | 存储系统内所有竞赛、作业规则以及参赛报名信息。 | 教师或管理员录入的比赛排期及配置数据、学生报名。 | 比赛基本信息、比赛关联题目、参赛选手列表。 | `id` (INT, PK), `title` (VARCHAR), `rules_type` (VARCHAR), `start_time` (DATETIME), `end_time` (DATETIME), `problems_json` (VARCHAR) |
| **D6** | 公告库 | MySQL (Notice表) | 存储系统首页渲染的系统公告、更新说明与维护通知。 | 管理员或教师编辑发布的公告文本内容。 | 首页展示的公告列表及具体内容。 | `id` (INT, PK), `title` (VARCHAR), `content` (TEXT), `author_id` (INT), `created_at` (DATETIME) |
| **Logs** | 系统审计日志库 | MySQL / 本地日志文件 | 存储管理员及关键角色的敏感系统操作日志，用于合规性审计。 | 拦截器/切面自动记录的操作明细。 | 安全审计列表。 | `id` (INT, PK), `user_id` (INT), `action` (VARCHAR), `ip_address` (VARCHAR), `created_at` (DATETIME) |

## 5.3 数据处理字典 (Data Process Dictionary)

数据处理字典定义了系统内各个二级子处理过程的逻辑关系与核心功能。

### 5.3.1 模块 1.0：用户注册与身份认证

*   **P1.1 用户注册**
    *   **输入**：F01 注册信息。
    *   **输出**：注册结果反馈（至 E1）。
    *   **处理逻辑**：读取 `D1 用户库` 校验用户名是否唯一；如无冲突，使用 BCrypt 算法对密码进行非对称哈希散列，并插入 `D1` 生成新用户记录。
*   **P1.2 用户登录与鉴权**
    *   **输入**：F01 登录凭证（账号、密码）。
    *   **输出**：F02 登录凭证 JWT。
    *   **处理逻辑**：从 `D1 用户库` 读取指定账号密码，利用 `BCrypt.checkpw()` 进行匹配；校验成功后根据用户角色组装 Token，进行 HMAC-SHA256 签名后返回。
*   **P1.3 JWT校验与拦截**
    *   **输入**：携带 JWT 的客户端 API 请求。
    *   **输出**：允许继续路由分发 / 鉴权失效拦截提示（401/403）。
    *   **处理逻辑**：提取 HTTP Header 中的 `Authorization` 字段，解析 JWT 签名；检查过期时间与权限声明，合法则解耦出 `userId` 并放入 ThreadLocal 请求上下文。
*   **P1.4 用户与权限管理**
    *   **输入**：E3 管理员的权限配置指令。
    *   **输出**：用户角色更改反馈。
    *   **处理逻辑**：根据指定的修改项，更新 `D1 用户库` 中用户的 `role`（如升级为 TEACHER）或锁定标志位。

### 5.3.2 模块 2.0：题库与测试数据管理

*   **P2.1 题目检索与分类过滤**
    *   **输入**：多条件查询参数（难度、标签、关键词）。
    *   **输出**：结构化题目列表（至 E1/E2）。
    *   **处理逻辑**：向 `D2 题目库` 发起 `SELECT` 复合查询，过滤不公开的题目，并实时结合用户的做题状态返回带 AC/WA 标记的题目列表。
*   **P2.2 题面详情加载**
    *   **输入**：题目 ID。
    *   **输出**：题面文本及样例（至 E1）。
    *   **处理逻辑**：单表读取 `D2 题目库` 的题目描述（Markdown格式）、时空限制参数以及前端可展示的静态输入输出样例。
*   **P2.3 题目信息维护**
    *   **输入**：F03 题目管理数据。
    *   **输出**：维护结果反馈（至 E2/E3）。
    *   **处理逻辑**：由教师或管理员操作，对 `D2 题目库` 进行题目创建、题面内容富文本更新、或软删除操作。
*   **P2.4 测试数据上传与存储**
    *   **输入**：F04 压缩用例包。
    *   **输出**：用例文件路径及状态（至 E2）。
    *   **处理逻辑**：解压上传的 zip 文件，校验输入输出文件是否一一对应（后缀配对）；将用例拷贝至本地 `D4 测试数据仓库`（TestCase目录），并往 `D2 题目库` 写入该题用例数量和映射路径。

### 5.3.3 模块 3.0：代码提交与成绩排行

*   **P3.1 代码在线编辑与自测**
    *   **输入**：在线编辑器中的源代码、语言类型、临时自测输入。
    *   **输出**：控制台自测执行结论（至 E1）。
    *   **处理逻辑**：不通过消息队列，直接发起单点沙箱调用，将输入的源代码与用户自定义数据送入沙箱运行，返回控制台 stdout/stderr，此过程不进行任何持久化。
*   **P3.2 代码提交与初始化**
    *   **输入**：F06 代码提交数据。
    *   **输出**：F07 初始提交记录、提交通知、排队反馈（至 E1）。
    *   **处理逻辑**：检查单次代码大小（限制 64KB），在 `D3 提交记录库` 插入状态为 `PENDING` 的行记录，并将该任务编号投递到后台判题队列或发布异步评测事件。
*   **P3.3 做题历史与日志检索**
    *   **输入**：查询参数（用户ID / 题目ID）。
    *   **输出**：做题历史列表、编译报错详情（至 E1/E2）。
    *   **处理逻辑**：分页读取 `D3 提交记录库` 中的评测条目，如状态为 `COMPILE_ERROR` 则一并读取编译报错日志。
*   **P3.4 成绩排行实时计算**
    *   **输入**：排行榜请求。
    *   **输出**：排行数据列表（至 E1）。
    *   **处理逻辑**：读取 `D1 用户库` 中已注册用户，按照 `solved_count` 降序、`submit_count` 降序计算前 N 名，输出并缓存结果以应对高并发访问。

### 5.3.4 模块 4.0：比赛与公告管理

*   **P4.1 比赛信息维护**
    *   **输入**：F17 比赛配置数据。
    *   **输出**：写规则反馈。
    *   **处理逻辑**：解析配置参数，保存比赛元数据、赛制（如 ACM 规则或 IOI 百分制）、参赛班级与比赛题目的绑定关系到 `D5 比赛库`。
*   **P4.2 比赛报名与状态控制**
    *   **输入**：学生报名请求 / 进入赛事请求。
    *   **输出**：参赛授权提示。
    *   **处理逻辑**：校验报名时间范围及白名单权限；若通过，在 `D5 比赛库` 的关系表中建立绑定，在比赛开始时刻激活答题倒计时。
*   **P4.3 比赛排行榜实时计算**
    *   **输入**：查询比赛榜单请求。
    *   **输出**：F18 比赛实时榜单。
    *   **处理逻辑**：从 `D5` 确定参赛人名单，从 `D3 提交记录库` 过滤出该时间段内各选手的有效提交；若为 ACM 规则，对 WA 提交累加 20 分钟罚时，并根据 AC 题数与罚时升序计算排行。
*   **P4.4 公告发布与展示**
    *   **输入**：公告编辑内容（E3）/ 获取公告（E1）。
    *   **输出**：有效公告通知流。
    *   **处理逻辑**：将编辑好的公告持久化到 `D6 公告库`；获取公告时拉取最新的前 N 条有效公告进行展示。

### 5.3.5 模块 5.0：异步代码评测

*   **P5.1 判题队列与任务调度**
    *   **输入**：F08 待评测记录。
    *   **输出**：源码及评测元数据。
    *   **处理逻辑**：判题后台守护线程无限轮询读取 `D3` 处于 `PENDING` 的行，或通过 MQ 消费者接收，放入本地线程池队列进行排队分发。
*   **P5.2 代码安全编译**
    *   **输入**：源码与指定语言。
    *   **输出**：编译出的二进制/字节码路径，若失败则输出编译错误（CE）。
    *   **处理逻辑**：在安全沙箱指定的挂载目录调用宿主机对应的编译器进行处理，若编译失败，截获错误控制台输出，直接发往 5.6。
*   **P5.3 安全沙箱运行**
    *   **输入**：二进制路径、时空限制（F09）、测试点输入（F10）。
    *   **输出**：程序输出数据（F12）、时空开销指标。
    *   **处理逻辑**：在 Linux CGroups 或 Windows Job Object 限制的进程环境下运行该可执行文件；实时监控 CPU 时间、最大 resident 内存使用；若超时超内存，立刻强制强杀并返回异常标记。
*   **P5.4 测试用例加载**
    *   **输入**：题目 ID。
    *   **输出**：测试点输入数据（F10）与期望输出数据（F11）。
    *   **处理逻辑**：根据题目 ID，从物理磁盘的 `D4 测试数据仓库` 读取所有后缀为 `.in` 和 `.out` 的对应文件，为每个用例依次执行迭代，逐一加载给沙箱运行。
*   **P5.5 输出结果比对**
    *   **输入**：实际输出（F12）、期望输出（F11）。
    *   **输出**：F13 单点评测状态。
    *   **处理逻辑**：对程序的实际 stdout 字节流与标准 `.out` 文件进行比对（默认使用 PE 过滤模式，自动移除末尾的换行符与连续空白字符）；若不一致，返回 `Wrong Answer`，并终止后续测试点判题（以节省资源）。
*   **P5.6 评测状态生成与更新**
    *   **输入**：汇总的评测状态、时空开销。
    *   **输出**：F14 评测结果与指标、F15 统计更新数据。
    *   **处理逻辑**：将该次提交在 `D3` 的状态改写为最终判定（AC/WA/RE 等），并回写最大耗时和内存；若此题为用户首次 AC，则在 `D1` 对该用户的 `solved_count` 加一，在 `D2` 对该题的 `accepted_count` 加一。

### 5.3.6 模块 6.0：系统监控与数据统计

*   **P6.1 系统资源监控**
    *   **输入**：物理机/JVM 的指标采样。
    *   **输出**：F16 系统监控数据。
    *   **处理逻辑**：通过 Java 管理接口（MXBean）及宿主机系统 API，采集 CPU 占比、物理剩余内存、JVM 堆大小、数据库连接池空闲数以及判题队列阻塞大小，定时向管理员端推送。
*   **P6.2 安全与行为审计**
    *   **输入**：管理员操作审计请求。
    *   **输出**：F20 操作审计日志。
    *   **处理逻辑**：从 `Logs 审计日志库` 中检索关键操作记录，按操作人、IP、时间范围进行多条件组合查询，以备溯源。
*   **P6.3 运营数据统计分析**
    *   **输入**：运营统计分析请求。
    *   **输出**：用户增长图表、提交趋势分析。
    *   **处理逻辑**：异步在 `D1`（用户总数）、`D2`（题库通过率分布）、`D3`（按天聚合的提交量曲线）中发起聚合查询，生成 JSON 数据返回前端图表插件（如 ECharts）渲染。
